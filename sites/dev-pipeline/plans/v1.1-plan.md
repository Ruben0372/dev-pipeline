# Dev Pipeline v1.1 -- Implementation Plan

**Project:** Dev Pipeline (methodology + system for shipping projects with Claude Code)
**Version:** v1.1
**Date:** 2026-03-26
**Baseline:** v1.0 (8-stage loop, site records, Notion sync, The Tower integration)
**Timeline:** 4 phases over ~2 weeks
**Driver:** Performance review findings from DELETE site feature benchmark (2026-03-26)

---

## Overview

v1.0 proved the pipeline ships code projects fast (33 min for a full CRUD feature with TDD). v1.1 hardens the pipeline for reliability, automates the manual overhead, and expands coverage beyond code projects. The guiding principle: the pipeline should work for everything Ruben ships -- Go services, Python tools, video projects, and enterprise software.

---

## Phase 1: Reliability (P0)

**Goal:** Eliminate the failure modes that wasted time during the v1.0 benchmark.
**Effort:** ~3 hours
**Dependencies:** None -- can start immediately.

### 1.1 Broken Window Gate

**Problem:** `TestNew_SchemaVersionIncremented` was broken since v1.1.0. The pipeline shipped on top of it.

**Solution:** Add a pre-ship gate that blocks deploys when pre-existing test failures exist.

| Task | Detail | Effort |
|------|--------|--------|
| Catalog existing failures | Run `go test ./...` and `npm test` across all active sites, record known failures | 30 min |
| Fix or skip with issue | Each failure: fix it, or mark `t.Skip("tracked: #issue")` with a linked Notion task | 30 min |
| Add `make test-gate` target | Runs full test suite, exits non-zero on any failure. `make ship` depends on it. | 15 min |

**Acceptance:** `make ship` refuses to proceed if any test fails. Zero skipped tests without a tracked issue.

### 1.2 Post-Deploy Smoke Check

**Problem:** Health endpoint returned 200 but no one verified the UI actually rendered.

**Solution:** `make smoke` target that curls key endpoints after deploy.

| Task | Detail | Effort |
|------|--------|--------|
| Define smoke endpoints per site | Each site's `operations/deployment.md` gets a `## Smoke Endpoints` section listing URL + expected status + expected body substring | 15 min |
| Implement `make smoke` | Bash script: reads endpoints from a `.smoke` file, curls each, checks status code and body grep, reports pass/fail | 30 min |
| Wire into deploy flow | `make deploy` calls `make smoke` after container comes up. Fail = roll back (print instructions, don't auto-rollback in v1.1) | 15 min |

**Acceptance:** `make smoke` runs after every deploy. Catches a down frontend or broken API route.

### 1.3 Docker Build Cache Warming

**Problem:** Alpine TLS flake on Arch wasted 15+ minutes. Cold pulls from Alpine mirrors occasionally fail.

**Solution:** Cron job on Arch that pre-warms Docker layer cache.

| Task | Detail | Effort |
|------|--------|--------|
| Create `scripts/cache-warm.sh` | Pulls base images (`golang:1.23-alpine`, `node:22-alpine`, `alpine:3.19`) and runs a no-op build to populate layer cache | 15 min |
| Schedule via The Tower | Create scheduled task: daily at 04:00, runs `cache-warm.sh` on Arch | 15 min |
| Pin Alpine mirror (fallback) | Add `RUN echo 'https://dl-cdn.alpinelinux.org/alpine/v3.19/main' > /etc/apk/repositories` to Dockerfiles as fallback if cache miss | 15 min |

**Acceptance:** Docker builds on Arch complete without TLS errors for 7 consecutive days.

## Phase 2: Automation (P1)

**Goal:** Remove manual steps from the pipeline loop. The pipeline should track itself.
**Effort:** ~10 hours
**Dependencies:** Phase 1 (smoke check and test gate must exist before automating around them).

### 2.1 Persistent Project Memory (Beads)

**Problem:** Mid-session, Claude had to discover `~/dashboard-deploy/` existed. Every session starts cold.

**Solution:** A `.beads` file per project that Claude Code loads at session start, containing deploy path, env vars, infra details, and known gotchas.

| Task | Detail | Effort |
|------|--------|--------|
| Define bead schema | YAML file: `project`, `deploy_path`, `env_file`, `docker_compose`, `ports`, `arch_ip`, `known_issues[]`, `smoke_url` | 30 min |
| Create beads for active projects | Write `.beads` for Dashboard, Dev Pipeline, Atlax, MilEats, Mentalist, Adune | 1 hr |
| Auto-load hook | Claude Code `PreToolUse` hook on `Bash`: if cwd is a project root with `.beads`, inject bead context into system prompt | 2 hrs |
| Bead update on deploy | After successful `make ship`, prompt to update `.beads` if any infra details changed | 30 min |

**Acceptance:** Starting a Claude Code session in any project root automatically loads deploy path, ports, and env var locations without manual discovery.

### 2.2 Semi-Automated Pipeline State Transitions

**Problem:** Pipeline stage (Ideate/Plan/Build/etc.) is updated manually. Easy to forget.

**Solution:** Claude Code hooks that detect stage-relevant actions and suggest advancing the pipeline.

| Task | Detail | Effort |
|------|--------|--------|
| Map triggers to stages | Define which actions signal stage completion: commit = Build done, `make test` pass = Test done, code-reviewer agent completes = Review done, `make deploy` succeeds = Ship done | 30 min |
| Implement `Stop` hook | On session end: check git log for commits since last stage change, check for test results, suggest next stage | 2 hrs |
| Notion sync on transition | When stage advances, update the project's Notion entry (Status property) and create a timestamped history entry | 1 hr |
| Manual override | `pipeline advance <stage>` and `pipeline set <stage>` commands for when automation guesses wrong | 30 min |

**Acceptance:** After committing code, Claude suggests "Build complete -- advance to Test?" without being asked. Notion stays in sync.

### 2.3 Notion Task Sync Hardening

**Problem:** Already improved to 2-min sync in v1.1.1, but sync failures are silent.

**Solution:** Add retry logic and failure alerting.

| Task | Detail | Effort |
|------|--------|--------|
| Add retry with backoff | 3 retries, exponential backoff (2s, 4s, 8s) on Notion API failures | 30 min |
| Failure logging | Log sync failures to `~/.dev-pipeline/sync.log` with timestamp and error | 15 min |
| Stale detection | If last successful sync > 10 min ago, warn at session start | 15 min |

**Acceptance:** Notion sync recovers from transient API failures. Stale state is surfaced, not hidden.

## Phase 3: Multi-Project Profiles (P2)

**Goal:** Make the pipeline fit every project Ruben works on, not just Go+React Docker services.
**Effort:** ~6 hours
**Dependencies:** Phase 2 (beads provide the project metadata profiles reference).

### 3.1 Profile Definitions

Four profile types, each defining which stages exist and what happens at each.

| Profile | Stages | Ship Means | Target Projects |
|---------|--------|------------|-----------------|
| **code** (default) | Ideate, Plan, Setup, Build, Test, Review, Ship, Retro | Docker deploy to Arch | Dashboard, Atlax, MilEats |
| **library** | Ideate, Plan, Build, Test, Review, Publish, Retro | `poetry publish` / `npm publish` / `go install` | Mentalist |
| **content** | Ideate, Plan, Shoot, Edit, Render, Review, Ship, Retro | Export + upload to platform | Adune |
| **enterprise** | Ideate, Plan, Setup, Build, Test, Review, **Compliance**, Ship, Retro | CDK deploy with migration review | Vitalis |

| Task | Detail | Effort |
|------|--------|--------|
| Define profile schema | YAML: `name`, `stages[]` (each with `name`, `trigger`, `gate`, `next`), `ship_action` | 1 hr |
| Implement 4 profiles | Write `profiles/code.yaml`, `library.yaml`, `content.yaml`, `enterprise.yaml` | 1 hr |
| Profile selection in scaffold | `scripts/scaffold.sh` asks for profile during `scaffold new <project>`, writes profile ref into `.beads` | 30 min |
| Profile-aware state machine | Hook logic from Phase 2.2 reads profile to determine valid transitions and stage names | 1 hr |

**Acceptance:** `scaffold new mentalist --profile library` creates a site record with Publish instead of Ship. Pipeline hooks respect the profile.

### 3.2 Stage Customization

**Problem:** Even within a profile, projects may need per-stage overrides (e.g., Vitalis needs HIPAA check in Compliance gate).

**Solution:** Per-site stage overrides in `.beads`.

| Task | Detail | Effort |
|------|--------|--------|
| Override schema | `.beads` gains `stage_overrides`: map of stage name to custom gate/action | 30 min |
| Compliance gate for enterprise | Implement checklist gate: blocks Ship until security-reviewer agent passes and migration plan exists | 1 hr |
| Registry publish for library | Implement publish action: detect package manager (poetry/npm/go), run appropriate publish command | 30 min |

**Acceptance:** Vitalis pipeline blocks at Compliance until HIPAA checklist passes. Mentalist pipeline publishes to PyPI at Publish stage.

## Phase 4: Observability (P3)

**Goal:** Measure pipeline performance so we can improve it with data, not intuition.
**Effort:** ~12 hours
**Dependencies:** Phase 2 (state transitions must be tracked before we can measure them), Phase 3 (profiles define what stages to measure).

### 4.1 E2E Test Stage

**Problem:** No visual regression testing. UI bugs caught only by manual inspection post-deploy.

**Solution:** Playwright E2E tests for web projects, gated in the Test stage.

| Task | Detail | Effort |
|------|--------|--------|
| Playwright setup | Add Playwright to Dashboard and future web projects. Config: chromium only, headless, Docker-compatible | 2 hrs |
| Critical path tests | Write E2E tests for each web project's critical flows (Dashboard: login, create project, create note, deploy site record) | 4 hrs |
| Wire into `make test` | `make test` runs unit tests + E2E. `make test-unit` and `make test-e2e` available separately. E2E only runs for `code` profile projects with a `web/` directory | 1 hr |
| Screenshot comparison | Playwright `toHaveScreenshot()` for key pages. Baseline screenshots committed to repo | 1 hr |

**Acceptance:** `make test-e2e` catches a broken page layout before deploy. Screenshot baselines exist for Dashboard.

### 4.2 Pipeline Metrics

**Problem:** "33 minutes" was measured manually. No historical data to detect trends.

**Solution:** Structured stage timing logged to SQLite (or JSON file for v1.1 simplicity).

| Task | Detail | Effort |
|------|--------|--------|
| Metrics schema | JSON lines file: `{project, version, stage, started_at, completed_at, duration_min, agent_used, success}` | 15 min |
| Auto-record on transition | Phase 2.2 hooks write a metrics entry when stage advances | 30 min |
| Bottleneck report | `pipeline metrics <project>` command: prints average time per stage, highlights slowest stage, compares to previous version | 1 hr |
| The Tower integration | API endpoint to serve metrics. Tower UI shows per-site stage duration bars | 2 hrs |

**Acceptance:** After shipping 3 features, `pipeline metrics dashboard` shows average stage times and identifies the bottleneck.

### 4.3 Session Logs

**Problem:** Pipeline history entries exist but aren't linked to the Claude Code sessions that produced them.

**Solution:** Link session IDs to pipeline history.

| Task | Detail | Effort |
|------|--------|--------|
| Capture session ID | On session start, record session identifier (timestamp + hash) in `.beads` temp field | 15 min |
| Link to history | Each pipeline history entry (stage transition) includes `session_id` | 15 min |
| Session replay summary | `pipeline history <project> --sessions` shows which sessions contributed to each version | 30 min |

**Acceptance:** `pipeline history dashboard --sessions` shows that v1.1.1 was built across 2 sessions with links to their handoff docs.

   no deps                     needs test gate              needs beads
                                needs smoke check            needs state machine
                                        |

                                  needs state transitions
                                  needs profiles

Phases 1 is independent. Phase 2 depends on Phase 1. Phase 3 depends on Phase 2. Phase 4 depends on Phases 2 and 3.

Within phases, tasks are largely parallelizable (e.g., 1.1, 1.2, 1.3 can run concurrently).

## Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| Bead auto-load slows session start | MEDIUM | Lazy-load: parse `.beads` only when first Bash command runs, not at session init. Keep beads under 50 lines. |
| Hook false positives (suggesting wrong stage) | MEDIUM | Hooks only suggest, never auto-advance. Include confidence score. User confirms with y/n. |
| Profile sprawl (too many custom stages) | LOW | Cap at 4 profiles in v1.1. Custom stages only via `.beads` overrides, not new profile types. |
| Playwright flaky in Docker on Arch | HIGH | Run headless Chromium with `--no-sandbox --disable-gpu`. Pin Playwright version. Retry once on failure before marking red. |
| Metrics JSON file grows unbounded | LOW | Single-user pipeline. At ~200 bytes per entry, 10K entries = 2MB. Add rotation in v1.2 if needed. |
| Notion API rate limits on frequent sync | LOW | Already at 2-min interval. Backoff handles transient failures. If persistent, increase to 5-min. |
| Enterprise compliance gate blocks velocity | MEDIUM | Compliance gate has an `--override` flag for hotfixes, logged as exception. Reviewed in retro. |

## Open Questions

1. **Bead storage location:** `.beads` in project root (next to `.git`) or in `dev-pipeline/sites/<project>/`? Leaning project root for discoverability.
2. **Profile inheritance:** Should `enterprise` extend `code` (inherit stages, add Compliance) or be fully independent? Inheritance is cleaner but adds complexity.
3. **Content profile validation:** Adune stages (Shoot/Edit/Render) have no automated gates. Accept manual-only transitions for v1.1, or build asset-presence checks?
4. **Metrics visualization:** Build into The Tower (adds scope) or use a simple CLI report for v1.1?
