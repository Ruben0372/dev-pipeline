---

## Vision

Harden the dev-pipeline into a self-managing system. Agents report bugs without context-switching. The Tower dispatches fixes automatically. Reliability gates prevent broken deploys. Project memory eliminates cold starts. Multi-project profiles make the pipeline work for Go services, Python tools, video projects, and enterprise software alike. Observability closes the loop with timing data and E2E tests.

---

## Decision Record

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Dispatch approval | Severity-dependent | Critical/High: auto-fix. Medium/Low: human approval. |
| Report storage | dev-pipeline `issues/` category | Reuses git-based sync, visible in Tower |
| Report mechanisms | Shell script + CLAUDE.md convention | Both ÔÇö script for automation, convention for any context |
| Scope | All pipeline projects | Not dashboard-specific |
| Priority interruption | Yes | Critical/High jump the dispatch queue |
| Beads location | Project root (`.beads`) | Discoverability > centralization |
| Profile inheritance | Independent | Simpler for v1.0.1, inheritance in v1.1 if needed |
| Content profile gates | Manual-only for v1.0.1 | Adune stages (Shoot/Edit/Render) have no automatable checks yet |
| Metrics storage | JSON lines file | Lightweight for single-user; SQLite in v1.1 if volume grows |

## Phase 1: Reliability Foundation

**Goal:** Eliminate failure modes from v1.0 benchmark. No broken deploys, no TLS flakes, no silent test failures.
**Effort:** ~3 hrs | **Deps:** None | **GitHub Issue:** #18

### 1.1 Broken Window Gate

**Problem:** `TestNew_SchemaVersionIncremented` was broken since v1.1.0. Pipeline shipped on top of it.

| Task | Detail |
|------|--------|
| Catalog existing failures | Run `go test ./...` and `npm test` across all active sites, record known failures |
| Fix or skip with issue | Each failure: fix it, or `t.Skip("tracked: #issue")` with linked GitHub issue |
| Add `make test-gate` | Full test suite, exits non-zero on any failure |
| Add `make ship` | Depends on `test-gate`. Blocks deploy on failures. |

### 1.2 Post-Deploy Smoke Check

**Problem:** Health endpoint returned 200 but no one verified UI rendered.

| Task | Detail |
|------|--------|
| Define smoke endpoints per site | Each site's `operations/` gets `## Smoke Endpoints` section |
| Implement `make smoke` | Reads `.smoke` file, curls each URL, checks status + body substring |
| Wire into deploy | `make deploy` calls `make smoke` after container up. Fail = print rollback steps. |

### 1.3 Docker Build Cache Warming

**Problem:** Alpine TLS flake on Arch wasted 15+ minutes.

| Task | Detail |
|------|--------|
| Create `scripts/cache-warm.sh` | Pulls base images, no-op build to populate layer cache |
| Schedule via Tower | Scheduled task: daily 04:00, runs cache-warm.sh on Arch |
| Pin Alpine mirror fallback | Add fallback mirror URL in Dockerfiles |

**Acceptance:**
- `make ship` blocks on test failures (zero broken windows)
- `make smoke` validates UI + API after every deploy
- Docker builds on Arch succeed without TLS flakes for 7 days

## Phase 2: Issue Reporting System

**Goal:** Agents can file structured reports from any context without changing what they're doing.
**Effort:** ~2 hrs | **Deps:** None | **GitHub Issues:** #12, #13

### 2.1 Report Format & Template

| Task | Detail |
|------|--------|
| Create `templates/issue-report.md` | YAML frontmatter schema: id, severity, domain, status, reporter, source_context, project, filed_at, dispatch_mode, assigned_to, resolved_at, dispatch_id |
| Add `issues` to `config/categories.json` | `{"label": "Issues", "order": 11, "icon": "alert-circle"}` |
| Create `config/dispatch-rules.json` | Maps severity Ôćĺ dispatch behavior (auto vs needs_approval) |

### 2.2 Shell Script & Auto-Push

| Task | Detail |
|------|--------|
| Create `scripts/report-issue.sh` | Accepts project, severity, domain, title, body. Generates frontmatter, writes file, commits. |
| Create `.githooks/post-commit` | Auto-push to origin in background |
| Configure hooks path | `git config core.hooksPath .githooks` |

### 2.3 CLAUDE.md Convention

Add to all project CLAUDE.md files:

```
## Issue Reporting
When you discover a bug outside your current domain:
1. DO NOT fix it or change context
2. Run: ~/projects/dev-pipeline/scripts/report-issue.sh {project} \
     --severity {critical|high|medium|low} \
     --domain {frontend|backend|infra|data|devops} \
     --title "Short description" --body "Details"
3. Continue your current work
```

**Acceptance:**
- One-liner files a report that appears in GitHub within seconds
- Post-commit hook auto-pushes without blocking
- Convention documented in all active project CLAUDE.md files

## Phase 3: Pipeline Automation

**Goal:** Remove manual overhead. Sessions start warm. Stages advance automatically. Sync is resilient.
**Effort:** ~10 hrs | **Deps:** Phase 1 | **GitHub Issue:** #19

### 3.1 Persistent Project Memory (Beads)

**Problem:** Every Claude session starts cold. Agent re-discovers deploy paths, ports, infra details.

| Task | Detail |
|------|--------|
| Define bead schema | YAML: `project`, `deploy_path`, `env_file`, `docker_compose`, `ports`, `arch_ip`, `known_issues[]`, `smoke_url`, `profile` |
| Create `.beads` for active projects | Dashboard, Dev Pipeline, Atlax, MilEats, Mentalist, Adune |
| Auto-load hook | `PreToolUse` hook on `Bash`: if cwd has `.beads`, inject context |
| Bead update on deploy | After `make ship`, prompt to update `.beads` if infra changed |

### 3.2 Semi-Automated State Transitions

**Problem:** Pipeline stage updated manually. Easy to forget.

| Task | Detail |
|------|--------|
| Map triggers to stages | commit = Build done, test pass = Test done, code-reviewer = Review done, deploy = Ship done |
| Implement `Stop` hook | On session end: check git log, suggest next stage |
| Notion sync on transition | Stage advance Ôćĺ update Notion project Status + history entry |
| Manual override | `pipeline advance <stage>` and `pipeline set <stage>` commands |

### 3.3 Notion Sync Hardening

| Task | Detail |
|------|--------|
| Retry with backoff | 3 retries, exponential (2s, 4s, 8s) on Notion API failures |
| Failure logging | Log to `~/.dev-pipeline/sync.log` with timestamp + error |
| Stale detection | If last sync > 10 min, warn at session start |

**Acceptance:**
- Sessions auto-load project context from `.beads`
- After committing code, Claude suggests "Build complete ÔÇö advance to Test?"
- Notion stays in sync with pipeline state
- Sync failures retry and surface, not swallowed silently

## Phase 5: Dispatch & Execution Engine

**Goal:** Wire dispatch to actually execute work. Approval flow for medium/low severity. Claude Code CLI for auto-fix.
**Effort:** ~12 hrs | **Deps:** Phase 4 | **GitHub Issues:** #15, #16

### 5.1 Dispatch Integration

| Task | Detail |
|------|--------|
| DB migration | Add `source_issue_id TEXT`, `approval_status TEXT DEFAULT 'auto'` to dispatches |
| `PUT /api/dispatch/{id}/approve` | Approve a needs_approval dispatch |
| `PUT /api/dispatch/{id}/reject` | Reject Ôćĺ sets issue to `wont_fix` |
| `IssueScanner` service | Scans `sites/*/issues/*.md` for `status: open`, creates dispatches |
| Wire into scheduler | Runs after each git pull |
| Severity routing | critical/high Ôćĺ `approved`. medium/low Ôćĺ `needs_approval` |
| Dedup | Check `source_issue_id` before creating dispatch |

### 5.2 Execution Engine

**Prereq:** Claude Code CLI installed on Arch server.

| Task | Detail |
|------|--------|
| `DispatchExecutor` service | Wraps Claude CLI invocation |
| Execution flow | Read issue Ôćĺ build prompt with project CLAUDE.md Ôćĺ invoke CLI Ôćĺ stream output Ôćĺ update issue status Ôćĺ push to dev-pipeline |
| Concurrency | One dispatch at a time (mutex), FIFO queue |
| Timeout | Configurable max execution time, kill on timeout |
| Safety | Critical/High fixes create branch + PR, not direct commit |
| Retry cap | Max 3 per issue + cooldown to prevent runaway loops |
| WebSocket | `dispatch_progress` (streaming), `dispatch_completed` |

**Acceptance:**
- Filing critical issue Ôćĺ auto-dispatched and fixed within 5 minutes
- Medium issue Ôćĺ queued, user approves in Tower, fixed within 5 min of approval
- No duplicate dispatches, no runaway loops
- Full lifecycle: open Ôćĺ dispatched Ôćĺ in_progress Ôćĺ resolved

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Auto-fix regressions | High | Critical/High fixes Ôćĺ branch+PR, not direct commit |
| Runaway dispatch loop | High | Max 3 retries per issue + cooldown |
| Git conflicts (concurrent agents) | Medium | Mutex on git ops (exists in GitSyncService) |
| Bead auto-load slows sessions | Medium | Lazy-load: parse on first Bash command, not session init |
| Hook false positives (wrong stage) | Medium | Hooks suggest only, never auto-advance. User confirms. |
| Playwright flaky in Docker on Arch | High | `--no-sandbox --disable-gpu`, pin version, retry once |
| Profile sprawl | Low | Cap at 4 profiles. Custom stages via `.beads` overrides only. |
| Large issue backlog | Low | Max concurrent dispatches, priority ordering |
| Claude CLI not on Arch | Medium | Phase 5 prereq: install Claude Code on Arch |
| Alpine TLS flakes | Medium | Phase 1 cache warming + mirror pinning |
