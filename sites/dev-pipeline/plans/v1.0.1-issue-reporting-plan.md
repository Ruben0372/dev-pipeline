# v1.0.1 — Pipeline Issue Reporting & Auto-Dispatch

> Dev-pipeline version bump from v1.0.0 → v1.0.1
> GitHub Project: #2 — "v1.0.1 — Pipeline Issue Reporting & Auto-Dispatch"

## Vision

Agents working on any project in the pipeline can report bugs, tasks, and improvements they discover — even when it's outside their current domain — without changing context. Reports flow into the Tower, are visible to the user, and are automatically dispatched to other agents for resolution based on severity rules.

## Decision Record

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Dispatch approval model | Severity-dependent | Critical/High: auto-dispatch + auto-fix. Medium/Low: auto-dispatch + human approval before execution. |
| Report storage | dev-pipeline `issues/` category | Reuses existing git-based sync, visible in Tower immediately via site records |
| Report mechanisms | Shell script + CLAUDE.md convention | Script for automation, convention for any agent context |
| Scope | All pipeline projects | Not dashboard-specific — any site in dev-pipeline |
| Priority interruption | Yes | High/Critical issues jump the dispatch queue |

## Phases

### Phase 1: Report Format & Convention (no code)

**Goal:** Define the standard and make it usable by agents immediately.

Tasks:
1. Create report template at `dev-pipeline/templates/issue-report.md`
2. Add `issues` category to `config/categories.json` (order: 11, icon: alert-circle)
3. Define severity → dispatch routing rules in a config file
4. Write CLAUDE.md convention section for all projects

**Deps:** None
**Estimate:** 1 session

### Phase 2: Shell Script & Post-Commit Hook

**Goal:** Give agents and humans a one-liner to file reports.

Tasks:
1. Create `scripts/report-issue.sh` — generates frontmatter, writes file, commits
2. Create `.githooks/post-commit` — auto-push to origin after every commit
3. Configure dev-pipeline repo: `git config core.hooksPath .githooks`
4. Test: file a report from CLI, verify it appears in GitHub

**Deps:** Phase 1
**Estimate:** 1 session

### Phase 3: Tower Issues Tab (Frontend)

**Goal:** Reports visible in the Tower with structured UI.

Tasks:
1. Add YAML frontmatter parser to `SiteRecordStore.Get()` — extract metadata
2. Create `IssueRecord` model (extends SiteRecord with frontmatter fields)
3. Add `GET /api/sites/{slug}/issues` endpoint — returns parsed issues with metadata
4. Create `IssueBoard.tsx` — Kanban columns: Open | Dispatched | In Progress | Resolved
5. Create `IssueCard.tsx` — severity badge, domain tag, age, assignee
6. Add Issues tab to `SiteRecordTabs.tsx` with special rendering (Kanban, not markdown list)
7. WebSocket listener for `issue_discovered` → re-fetch + toast notification

**Deps:** Phase 2
**Estimate:** 2 sessions

### Phase 4: Dispatch Integration (Backend)

**Goal:** Wire the dispatch service to actually execute work.

Tasks:
1. Enhance `Dispatch` model — add `source_issue_id`, `approval_status` fields
2. Add migration: `ALTER TABLE dispatches ADD COLUMN source_issue_id TEXT, approval_status TEXT DEFAULT 'auto'`
3. Add `PUT /api/dispatch/{id}/approve` endpoint
4. Add `PUT /api/dispatch/{id}/reject` endpoint
5. Create `IssueScanner` service — scans `sites/*/issues/*.md` for `status: open`
6. Wire IssueScanner into scheduler (runs after each git pull)
7. Routing logic:
   - `critical`/`high` severity → create dispatch with `approval_status: approved`
   - `medium`/`low` severity → create dispatch with `approval_status: needs_approval`
8. Update `processDispatch()` — skip `needs_approval` items, only process `approved`
9. Link dispatch back to issue: on creation, update issue frontmatter `dispatch_id`

**Deps:** Phase 3
**Estimate:** 2 sessions

### Phase 5: Auto-Dispatch Execution Engine

**Goal:** Dispatches actually invoke Claude Code to fix issues.

Tasks:
1. Create `DispatchExecutor` service — wraps `claude` CLI invocation
2. Execution flow:
   a. Read issue content → build prompt with project context
   b. Invoke `claude --print --dangerously-skip-permissions` (or structured API)
   c. Stream output to dispatch record
   d. On success: update issue `status: resolved`, `resolved_at`
   e. Push updated issue to dev-pipeline
   f. Log to activity timeline
3. Concurrency control — one dispatch at a time (mutex), queue-based
4. Timeout handling — kill long-running dispatches after configurable limit
5. Error handling — on failure, set dispatch to `failed`, issue stays `open`
6. WebSocket updates during execution: `dispatch_progress`, `dispatch_completed`

**Deps:** Phase 4
**Estimate:** 2–3 sessions

### Phase 6: Feedback Loop & Polish

**Goal:** Full lifecycle visible in Tower, notifications, and cross-project support.

Tasks:
1. Activity timeline integration — filed/dispatched/resolved events
2. Notification badges — unread issue count on Issues tab
3. Issue metrics — KPI card: "X open issues across Y projects"
4. Cross-project dashboard — aggregate issues view from all pipeline sites
5. CLAUDE.md updates — all project CLAUDE.md files get the reporting convention
6. Documentation — update dev-pipeline README with issue reporting workflow
7. Version bump — dev-pipeline v1.0.1 tag

**Deps:** Phase 5
**Estimate:** 1–2 sessions

## Severity → Dispatch Routing Rules

| Severity | Dispatch | Approval | Auto-Fix | Queue Priority |
|----------|----------|----------|----------|----------------|
| Critical | Immediate | Auto-approved | Yes | Interrupts current queue |
| High | Immediate | Auto-approved | Yes | Next in queue |
| Medium | Queued | Needs human approval | After approval | Normal FIFO |
| Low | Queued | Needs human approval | After approval | Normal FIFO |

## CLAUDE.md Convention (Added to All Projects)

```markdown
## Issue Reporting

When you discover a bug, task, or improvement outside your current domain:

1. DO NOT fix it or change context
2. File a report:
   ```bash
   ~/projects/dev-pipeline/scripts/report-issue.sh {project-slug} \
     --severity {critical|high|medium|low} \
     --domain {frontend|backend|infra|data|devops} \
     --title "Short description" \
     --body "What you found, reproduction steps, suggested fix"
   ```
3. Continue your current work

The Tower will pick up the report, route it based on severity, and dispatch
an agent to handle it. Critical/High issues are auto-fixed. Medium/Low
require human approval in the Tower UI.
```

## New API Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | /api/sites/{slug}/issues | List parsed issues with frontmatter metadata |
| GET | /api/sites/{slug}/issues/{id} | Get single issue detail |
| PUT | /api/dispatch/{id}/approve | Approve a needs_approval dispatch |
| PUT | /api/dispatch/{id}/reject | Reject a dispatch (sets issue to wont_fix) |
| GET | /api/issues | Cross-project: all open issues across all sites |
| GET | /api/issues/stats | Counts by severity, status, project |

| Type | When | Payload |
|------|------|---------|
| `issue_discovered` | Scanner finds new open issue | `{project, id, severity, title}` |
| `issue_dispatched` | Dispatch created for issue | `{project, id, dispatch_id}` |
| `dispatch_progress` | Execution output streaming | `{dispatch_id, chunk}` |
| `dispatch_completed` | Fix applied | `{dispatch_id, issue_id, status}` |
| `issue_needs_approval` | Medium/Low issue queued | `{project, id, severity, title}` |

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Auto-fix introduces regressions | High | Critical/High fixes create branch + PR, not direct commit |
| Runaway dispatch loop (fix creates new issue) | High | Max 3 dispatch retries per issue, cooldown period |
| Git conflicts from concurrent agents | Medium | Mutex on git ops (already exists in GitSyncService) |
| Large issue backlog overwhelms queue | Low | Configurable max concurrent dispatches, priority ordering |
| Claude CLI not available on Arch | Medium | Phase 5 prereq: install Claude Code on Arch server |

## Success Criteria

1. Agent files report from CLI → appears in Tower Issues tab within 30s
2. Critical issue auto-dispatched and fixed within 5 minutes
3. Medium issue queued, user approves in Tower, fixed within 5 minutes of approval
4. Full lifecycle visible: filed → dispatched → in_progress → resolved
5. Zero context switches for the reporting agent
