## What Went Well

### 1. Parallel Research Agents

Before writing any code, parallel agents evaluated library options for each domain: Motion vs react-spring vs auto-animate, BlockNote vs TipTap vs Slate, Zustand vs Jotai vs Redux Toolkit. This front-loaded decision-making and prevented mid-build library switches. Each agent produced a comparison matrix with bundle size, API surface, and TypeScript support.

### 2. Phased Delivery with Testing Gates

The 5-phase structure (Zustand first, then animations/editor/timer in parallel, then integration) kept the codebase stable. All existing tests passed between phases. No phase was merged into the main flow until its tests were green. This prevented the cascading breakage seen in v1.0 when agents modified shared files simultaneously.

### 3. Zustand-First Architecture

Starting with Zustand stores before building features was the right call. Every subsequent phase (animations, timer, editor) could immediately use centralized state without prop drilling or late refactoring. The persist middleware gave timer state and theme state for free.

### 4. Code Splitting Strategy

Isolating BlockNote and Mantine into manual chunks kept non-/notes routes at v1.0.1 load speeds. The 2MB addition only affects users who navigate to the notes editor. This was validated by checking Vite build output and the browser network tab before deploying.

### 5. Dev-Pipeline as Integration Layer (v1.1.1)

Replacing the SQLite-backed site records with a git-backed filesystem reader was a clean architectural win. Agents write markdown files to `~/projects/dev-pipeline/sites/{project}/`, push to GitHub, and The Tower pulls and renders them automatically. No database migration, no sync logic, no API for writes. Git is the source of truth.

## What Didn't Go Well

### 1. Blank Screen from manualChunks Misconfiguration

The Vite `manualChunks` configuration for BlockNote caused a blank white screen in production. The chunk names collided with Vite's automatic splitting, resulting in missing module references at runtime. The development server worked fine because it doesn't use chunking -- the bug only manifested in `vite build` output served by Nginx. This required debugging the built assets directly.

**Root cause:** The manualChunks function was too aggressive, pulling transitive dependencies into named chunks that other chunks also expected to own.

**Fix:** Narrowed the chunk boundaries to only the top-level BlockNote and Mantine entry points, letting Vite handle transitive dependencies automatically.

**Time lost:** ~45 minutes of debugging a production-only issue.

### 2. Git safe.directory Issue on Arch

The git-pull scheduler in v1.1.1 failed silently because Docker runs as a different UID than the git repo owner on the host filesystem. Git's `safe.directory` check rejected the pull operation. The API logs showed no errors because the exec.Command exit code was swallowed.

**Root cause:** Git 2.45+ enforces ownership checks by default. The Docker container's user didn't match the repo's directory owner on the mounted volume.

**Fix:** Added `git config --global --add safe.directory /data/dev-pipeline` to the API Dockerfile's entrypoint. Also added error logging to `runGitPull()` so failures surface in the API logs.

**Time lost:** ~30 minutes, mostly spent SSH-ing into Arch and reading Docker logs.

### 3. Preview Hook False Positives

The preview verification step (screenshot + console log check after each phase) flagged false positives from Motion's layout animations. Initial render showed elements at opacity 0 or translated off-screen, which the automated check interpreted as missing content. Manual verification confirmed the elements animated in correctly after the initial frame.

**Root cause:** Screenshots captured the first paint frame before animations completed. Motion's AnimatePresence renders children at their initial state (e.g., opacity: 0) before animating to the target state.

**Impact:** Low. Added 10-15 minutes of false-alarm investigation across phases. The preview hook is still valuable -- it just needs a delay or "wait for idle" step before capturing.

## Lessons Learned

### 1. Production-Only Bugs Need a Production-Like CI Step

The manualChunks blank screen would have been caught by running `vite build && vite preview` as a CI gate. Development mode hides bundling issues. For any change to Vite's build configuration, always verify with a production build before deploying.

### 2. Docker UID Mismatches Are a Recurring Pattern

The safe.directory issue is the same class of problem as file permission errors on mounted volumes. Any time the API container interacts with host-owned files (SQLite DB, git repos, note files), UID alignment must be verified. Consider standardizing on a fixed UID in the Dockerfile that matches the Arch host user.

### 3. Animation Testing Requires Timing Awareness

Snapshot-based testing and screenshot verification both struggle with animated UIs. Options for v1.2+: (a) disable animations in test environment via `prefers-reduced-motion`, (b) add a configurable delay to preview screenshots, or (c) use Playwright's `waitForLoadState('networkidle')` before capturing.

### 4. Git as a Document Store Works for Single-User

The dev-pipeline pattern (agents write files, git syncs, Tower reads) is simple and effective for a single-user dashboard. It avoids the complexity of a document API, handles conflict-free writes (one author at a time), and provides version history for free. This pattern would not scale to multi-user without a merge strategy.

### 5. Zustand Stores Should Be Designed Before Features

Having the store interfaces defined in Phase 1 meant every subsequent phase had a clear contract for state access. This is the same lesson as v1.0's "plan before build" -- but applied to frontend state architecture specifically.

## What's Next: v1.2 and Beyond

### Persistent Memory (Beads)

Per-project context that auto-loads into Claude Code sessions. Deploy paths, API schemas, known issues, and infrastructure details persist as structured "beads" rather than relying on CLAUDE.md and handoff documents. This closes the gap where agents had to rediscover deployment paths at ship time.

### Lottie Animations

Replace static Lucide icons with Lottie animations for key interactions: loading states, success confirmations, empty states. Lottie-web is ~50KB and renders After Effects animations at 60fps. Evaluated but deferred from v1.1.0 to keep scope bounded.

### Full File Sync

Extend the dev-pipeline pattern to support bidirectional sync: Tower can create/edit documents that push back to the git repo. This requires a write API and conflict resolution strategy. Current read-only approach is stable and sufficient for now.

### Claude Code State Transition Hooks

Hook into Claude Code's session lifecycle: on session start, auto-load the project's site records as context. On session end, auto-commit any generated documents to dev-pipeline. On stage transition (e.g., Build to Test), update the project's status in The Tower. This turns the dev-pipeline from passive storage into an active orchestration layer.
