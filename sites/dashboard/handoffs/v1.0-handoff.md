**Duration:** Single session, ~6 phases implemented

---

## Purpose

This project was initiated to stress-test Ruben's 8-stage dev pipeline (`~/projects/ruben-dev-pipeline.html`). The Dashboard was chosen as a real deliverable that would exercise the pipeline end-to-end: from planning through deployment on real infrastructure (Arch server via Tailscale). This document captures what worked, what broke, and what the pipeline should look like going forward.

## Pipeline Efficiency Analysis

### What Worked Well (KEEP)

**1. Parallel agent architecture was the biggest win.**
Launching 2-3 agents simultaneously (e.g., backend + frontend + planner) cut wall-clock time dramatically. Phase 1 had a planner + React scaffold + Go/SQLite agent running in parallel. Phase 5 had backend + frontend agents. This is the pipeline's strongest pattern.

**2. Planner agent producing detailed implementation plans.**
The Phase 1 planner produced a 23-step plan with file paths, dependency graphs, risk mitigations, and success criteria. This made the build agents effective because they had unambiguous instructions. The plan acted as a contract between agents.

**3. TDD agents writing tests alongside implementation.**
Go tests (13 passing — config, handlers, database) and React tests (26 passing — Sidebar, Layout, DashboardPage, API client) were written by dedicated test agents. Coverage wasn't perfect (db at 66.7% due to hard-to-trigger error paths) but the pattern of test-alongside-build worked.

**4. Incremental deployment model.**
SSH into Arch → `git pull` → `docker compose up -d --build` is fast and reliable. The Docker multi-stage builds (Alpine for Go with CGO, Node for Vite, Nginx for serving) work well. Build caching means only changed layers rebuild.

**5. CLAUDE.md as living context.**
Having project structure, ports, host IPs, Notion database IDs, and preferences in CLAUDE.md meant every agent had the context it needed without re-discovery. This is essential for multi-agent workflows.

---

### What Failed or Fell Short (FIX)

**1. Agent conflicts on shared files — the #1 pain point.**
Multiple agents modifying `main.go`, `go.mod`, `go.sum`, and `App.tsx` simultaneously caused merge conflicts and overwrites. The Notion agent bumped Go from 1.23 to 1.25 without coordination. The Calendar agent and Notion agent both modified `main.go` with different import blocks.

> **Recommendation:** For shared files (main.go, App.tsx, go.mod), use a single "integration agent" that runs AFTER feature agents complete. Feature agents should create their packages/components in isolation and produce a "wiring instructions" output that the integration agent applies sequentially. or you can use git isolated worktrees

**2. Notion API property name mismatches.**
The sync code assumed property names like "Name", "Status", "Priority" but the actual Notion database used different names or property types (e.g., "status" type vs "select" type). This caused silent failures — the frontend showed "failed to load" for every field.

> **Recommendation:** Always fetch the Notion database schema (`notion-fetch` tool) before writing sync code. Include the actual schema in the agent's prompt. Add a schema validation step in Stage 3 (Setup).

**3. Database ID vs Data Source ID confusion.**
The `.env` file used the Notion *data source* ID (`collection://...`) instead of the *database* ID for `NOTION_PROJECTS_DB`. The Notion API's `/v1/databases/{id}/query` endpoint requires the database ID. This caused 404s.

> **Recommendation:** CLAUDE.md should clearly label which ID is which. The `.env.example` should have comments explaining the difference. Validation at server startup should verify the ID works before proceeding.

**4. Notes update bug — FTS5 trigger issue.**
Updating a note crashed because the FTS5 virtual table wasn't being updated correctly. The `UPDATE` path tried to delete + re-insert into `notes_fts` but the SQL was wrong. This required a dedicated fix.

> **Recommendation:** FTS5 operations need their own integration test that exercises create → search → update → search → delete → search. Add this to the test suite.

**5. SSH access wasn't pre-configured.**
Deploying to Arch was blocked because no SSH key existed on the Mac for the Arch server. The user had to generate keys, copy them, and test. This interrupted flow.

> **Recommendation:** Add SSH key verification to Stage 3 (Setup) checklist. Before any deployment step, verify: `ssh -i KEY USER@HOST "echo ok"`.

**6. No preview verification during development.**
Code was written, compiled, and deployed without visual verification until the very end. The preview server (`preview_start`) should have been used after each phase to catch layout issues, API errors, and rendering bugs early.

> **Recommendation:** Add a mandatory "preview checkpoint" after each phase in the pipeline. Use `preview_screenshot` + `preview_console_logs` to verify before moving on.

**7. Design iteration was expensive.**
The first design pass scored 4/10 from Ruben. The glassmorphism was too heavy, fonts too small, too many colored containers. It took a full redesign pass (60+ file changes) to get to the minimal CLI aesthetic he wanted. The design preferences form helped but was created too late.

> **Recommendation:** Run the design preference discovery BEFORE Phase 1. Add a "Design Intent" section to the PRD in Stage 2 (Plan). Include: font family, color philosophy (minimal vs vibrant), card style, density preference. This saves an entire redesign cycle.

---

### What to Discard

**1. The `init()` handler pattern.**
The Go handlers use `init()` functions to reassign package-level variables (`ListProjects`, `GetProject`, etc.). This is fragile, hard to test, and Go-unidiomatic. Replace with dependency injection via constructor functions or a handler struct.

**2. Simulated dispatch processing.**
The dispatch service has a fake 1-second "processing" step. This was a placeholder. Either connect it to a real task runner or remove the simulation — it creates false expectations.

**3. Google Fonts CDN dependency.**
The frontend loads IBM Plex Mono from Google Fonts CDN. For a self-hosted dashboard on a Tailscale network, this creates an external dependency. Bundle the font files locally.

## Recommendations for Pipeline v2

### Stage 2 (Plan) — Add Design Intent
Before any code is written, capture:
- Font family and size scale
- Color philosophy (minimal/vibrant/branded)
- Card/container style (flat, bordered, glass, none)
- Layout density (spacious vs compact)
- Reference sites or screenshots

### Stage 3 (Setup) — Add Infrastructure Checklist
- [ ] SSH access to deploy target verified
- [ ] `.env` file with ALL required values (not just some)
- [ ] Notion database schema fetched and documented
- [ ] Database IDs vs Data Source IDs clearly labeled
- [ ] Docker builds on deploy target verified

### Stage 4 (Build) — Agent Coordination Protocol
1. Feature agents create packages in isolation (no shared file edits)
2. Integration agent wires features into `main.go`, `App.tsx`, etc.
3. Preview checkpoint after each phase
4. No agent should modify `go.mod` version — only `go mod tidy`

### Stage 5 (Test) — Coverage Gates
- Unit tests: 90% (current pipeline rule, enforce it)
- Integration tests: API endpoints with real SQLite (in-memory)
- E2E tests: At minimum, health check + page navigation
- Add FTS5-specific test suite for notes

### Stage 6 (Review) — Run Code Review Agent
The `code-reviewer` agent was never invoked during this session. It should be mandatory before any commit. Same for `security-reviewer` on API endpoints handling user input.

### Stage 7 (Ship) — Formal Deploy Checklist
- [ ] All tests pass locally
- [ ] Docker builds clean on deploy target
- [ ] `.env` on deploy target matches `.env.example`
- [ ] Health endpoint returns 200 after deploy
- [ ] Rollback command documented (`git revert` + `docker compose up --build`)
- [ ] Smoke test: load each page, verify no console errors

---

## Key Takeaway

The pipeline works. Parallel agents are the killer feature — they turn a multi-day project into a single session. But the pipeline needs stronger gates between stages: design intent before building, schema validation before API integration, preview verification before deployment, and code review before committing. The biggest time sink was reactive debugging (wrong Notion IDs, FTS5 bugs, design rework) that could have been caught with upfront validation.

The 8-stage loop is solid. The weakness isn't the stages — it's the transitions between them.
